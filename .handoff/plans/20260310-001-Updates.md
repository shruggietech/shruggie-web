## Purpose and Context

This sprint plan defines the complete ground-up construction of the ShruggieTech public website (`https://shruggie.tech`). The project is a greenfield build with no predecessor codebase. The website specification (v1.2.0) is the authoritative reference for all architectural decisions, design system values, page content, component behavior, accessibility requirements, and deployment configuration. Every work item in this document cross-references the specification sections that govern its implementation.

The build produces a production-ready Next.js 15 (App Router) website deployed to Vercel, featuring: a branded design system with dark/light mode, Lenis smooth scrolling, Framer Motion scroll-triggered animations, an MDX-based blog and case study content pipeline, Formspree-powered contact form, four audience-specific landing pages, comprehensive SEO/AEO infrastructure with JSON-LD schema markup, dynamic Open Graph image generation, cookie-consent-gated analytics, WCAG 2.2 AA accessibility compliance, and full Core Web Vitals optimization.

The plan contains 12 work items organized across 5 phases. Phases represent logical construction stages; work items within a phase may be executed sequentially or in parallel depending on dependency analysis documented in the Implementation Ordering section below.

**Human-in-the-loop items:** Eight items require manual action by a ShruggieTech team member and cannot be produced by an implementing agent. These are catalogued in the specification at [§1.4](../../ShruggieTech_Website_Specification.md#14-human-in-the-loop-requirements). Of these, items 1 (logo PNG) and 8 (favicon source artwork) are required before development begins. Items 2, 3, 4, and 7 are required before launch but not before development. Items 5 and 6 (team photos, case study screenshots) are content items that render as placeholders until provided. Agents should implement placeholder rendering for all content-dependent assets.

```text
  Item 10 (Research + Products + Error Pages)  ── depends on Item 4

Phase 5: Infrastructure and Launch Preparation (sequential, final)
  Item 12 (SEO, Analytics, OG Images, Deployment) ── depends on ALL prior items
```

### Execution Mode Recommendations

| Item(s) | Recommended Mode | Rationale |
|---------|-----------------|-----------|
| 1 → 2 → 3 → 4 | Sequential | Each item builds directly on the output of the previous. No parallelism is possible. |
| 5, 6, 7 | Parallel Independent | Three agents in isolated worktrees. No shared file modifications. All three depend only on the Item 4 baseline. |
| 8 → 9 | Sequential | Item 9 reuses the MDX content pipeline established in Item 8. The `lib/blog.ts` patterns inform the case study loader. |
| 10, 11 | Parallel Independent | Two agents in isolated worktrees. No shared file modifications. Both depend only on the Item 4 baseline. Can run concurrently with Phase 3. |
| 12 | Sequential | Final integration item. Touches shared files (`app/layout.tsx`, `next.config.ts`, `lib/schema.ts`). Must verify against the cumulative codebase. |

### Branch Naming Convention

```text
sprint/20260310-001/<item-number>-<short-description>
```

After parallel execution phases, branches merge to `main` in numerical order. After each merge, the full development server must start without errors before proceeding to the next merge.

### 4. Global Layout and Navigation (Critical)

**Description:** Implement the root layout, Lenis smooth scrolling provider, site header (desktop and mobile), footer, mobile navigation overlay, and dark/light mode toggle. After this item, the site has a fully functional shell into which individual pages are rendered.

**Implementation Instructions:**

A. Create `components/layout/LenisProvider.tsx` — Client component. Initializes Lenis with the configuration values from the spec (duration 1.0, exponential decay easing, touchMultiplier 1.5). Checks `prefers-reduced-motion` at initialization and bails out entirely if active. Listens for `prefers-reduced-motion` changes during the session and destroys Lenis if the preference is activated mid-session. Copy the implementation from the spec.

B. Create `components/layout/Header.tsx` — Client component (needs scroll position state and theme toggle interactivity). Fixed position, full-width, `z-50`. Translucent background with backdrop-blur that progressively gains opacity as the user scrolls (0% at scroll 0, 80% at 100px). Contains: logo (kawaii shruggie PNG at 32px height, or placeholder if asset unavailable), wordmark ("ShruggieTech" in Space Grotesk Medium 18px), navigation links (Services, Work, Research, Products, About, Blog) as `<Link>` components with animated underline hover effect, theme toggle (Sun/Moon from Lucide with `aria-label`), and "Get in Touch" small primary button linking to `/contact`. On mobile (< 768px), replace nav links with a hamburger icon that triggers the MobileNav. Progressive opacity effect is disabled when `prefers-reduced-motion` is active.

C. Create `components/layout/Footer.tsx` — Server component. Four-column layout on desktop, stacked on mobile. Column 1: logo + tagline. Column 2: page links (Services, Work, Research, Blog, About, Contact). Column 3: product links (shruggie-indexer, metadexer, rustif). Column 4: GitHub link. Bottom bar: dynamic copyright year with `{new Date().getFullYear()}`, "Made in the USA 🇺🇸", and "Privacy Policy" link to `/privacy`. Footer uses `bg-bg-secondary` to create visual separation.

D. Create `components/layout/MobileNav.tsx` — Client component. Full-screen overlay sliding in from the right. Uses primary background at full opacity. Links in Space Grotesk Medium at 28px with 24px vertical spacing. Hamburger triggers open, X icon triggers close. Both buttons have `aria-expanded` and `aria-controls`. Focus is trapped within the overlay when open (Tab key does not escape). Pressing Escape closes it. Underlying scroll is locked via `document.body.style.overflow = "hidden"` (or Lenis `stop()`/`start()` if Lenis is active).

E. Assemble `app/layout.tsx` — The root layout wraps the entire application. It must: load all three fonts via `next/font/local` with the variable names `--font-display`, `--font-body`, `--font-mono`; apply the font variables and the `dark` class to the `<html>` tag; inject the theme initialization script via `dangerouslySetInnerHTML` from `getThemeScript()` inside a `<script>` tag in `<head>`; inject favicon and manifest metadata using the Next.js `Metadata` export (icons, manifest, themeColor arrays); wrap `{children}` in the `LenisProvider`; render `SkipLink` as the first element inside `<body>`; render `Header` after SkipLink; wrap `{children}` in `<main id="main-content">`; render `Footer` after main. Set the default `Metadata` export with `title` template (`"%s | ShruggieTech"`), default `description`, and `metadataBase`.

F. Implement the theme toggle logic. The toggle button in the Header sets a `theme` cookie with value `"light"` or `"dark"` and toggles the `dark` class on `document.documentElement`. The cookie has a 1-year expiry and `SameSite=Lax`. The `getThemeScript()` IIFE in the root layout reads this cookie on first paint, falling back to `prefers-color-scheme`, defaulting to dark.
